# SAA-C03：クラウドの迷宮を歩くエンジニアのための技術演劇集

## 登場人物
- **佐藤 (Sato)**: オンプレミス一筋15年。データセンターの冷気とラッキングの重みを愛するベテラン。最近、不本意ながらクラウド移行プロジェクトのリーダーに任命された。物理故障こそがエンジニアの「本番」だと思っている。
- **田中 (Tanaka)**: シニカルなクラウドネイティブ・エンジニア。物理サーバーを「場所を取る暖房器具」だと思っている。趣味はIAMポリシーの最小権限設定と、サーバーレス化による保守コストの削減。

---

## 第1幕：コンピューティングと高可用性 〜「サーバー」は家畜か、ペットか〜

### 【スキット】
**佐藤**: 「おい田中、さっきからEC2のインスタンスが1台勝手に消えたんだが。誰だ、勝手にラックから抜いた奴は。現場の作業連絡は入ってないぞ」
**田中**: 「佐藤さん、ここはデータセンターじゃないんです。Auto Scalingが『お前、最近やる気ないな』って判断してターミネートしただけですよ。死んだ魚の目をしているサーバーを介護するのは、オンプレ時代の悪習です」
**佐藤**: 「何を言う。あの中には職人芸でチューニングした設定ファイルが……」
**田中**: 「まだそんなこと言ってるんですか。設定はUserDataかAMIに焼き付けてください。サーバーは『名前をつけて可愛がるペット』じゃなくて『番号で管理する家畜』なんです。病気になったら殺して次を連れてくる。それがクラウドの慈悲ってやつですよ」
**佐藤**: 「……冷血だな。だが、ユーザーはどうやってその『新しい家畜』を見つけるんだ？」
**田中**: 「ELB（Elastic Load Balancing）ですよ。彼が玄関で交通整理をして、生きてる家畜にだけ仕事を振り分けます。あなたが深夜にパトランプを見て駆けつける代わりに、彼が無言で切り離してくれます」

### 【仕組み：どう動いているか】
AWSのコンピューティングは、物理的なハードウェアを抽象化した「仮想マシン（EC2）」をベースにしています。
- **Auto Scaling**: 負荷（CPU使用率など）やスケジュールに基づいて、インスタンスの数を自動的に増減させます。内部的にはCloudWatchアラームをトリガーに、EC2のAPIを叩いて起動・削除を行っています。
- **ELB**: クライアントからのリクエストを受け取り、ターゲットグループに登録された正常なインスタンスにトラフィックを分散します。ヘルスチェックを行い、応答がないインスタンスにはパケットを送らないようにします。

### 【詳細バリエーションと設定】
- **EC2購入オプション**:
    - **オンデマンド**: 1秒単位の支払い。短期・予測不能な負荷用。
    - **リザーブドインスタンス (RI)**: 1年または3年のコミット。定常的な負荷用。
    - **スポットインスタンス**: AWSの余剰キャパシティを利用。最大90%オフだが、AWS側の都合で中断される。
    - **Savings Plans**: 期間（1年または3年）のコミットで柔軟に割引が適用される。
- **ELBの種類**:
    - **ALB (Application Load Balancer)**: L7層（HTTP/HTTPS）。パスベース・ホストベースのルーティングが可能。
    - **NLB (Network Load Balancer)**: L4層（TCP/UDP）。超低レイテンシ、固定IPが必要な場合。
    - **GWLB (Gateway Load Balancer)**: サードパーティの仮想アプライアンス（ファイアウォール等）用。

## 第2幕：ストレージの迷宮 〜容量設計という名の呪縛からの解放〜

### 【スキット】
**佐藤**: 「田中、ストレージの見積もりがほしい。来期のデータ増加率を30%と見込んで、余裕を持って50TBのSANを……」
**田中**: 「佐藤さん、まだタイムスリップしてるんですか？ 令和の時代に『容量設計』なんて死語ですよ。S3を使ってください。容量は無限、耐久性は9が11個並ぶ変態的なスペックです」
**佐藤**: 「無限だと？ 予算はどうなる。上司に『無限です』なんて言ったら、私の首が物理的にターミネートされるぞ」
**田中**: 「使った分だけ払えばいいんです。アクセスしないデータは自動的に『S3 Glacier』っていう冷凍保存庫に送る設定にすれば、あなたの飲み代より安く済みます。あと、複数のサーバーでファイルを共有したいなら、NFSじゃなくてEFSを使ってください」
**佐藤**: 「……ディスクの回転音が聞こえないストレージなんて、信用できるのか？」

### 【仕組み：どう動いているか】
- **Amazon S3 (Simple Storage Service)**: オブジェクトストレージ。データは3つ以上の可用性ゾーン（AZ）に自動的に複製され、99.999999999%（イレブンナイン）の耐久性を実現します。
- **Amazon EBS (Elastic Block Store)**: EC2にアタッチするブロックストレージ。ネットワーク経由で接続されますが、物理的には同じAZ内のホストに配置されます。
- **Amazon EFS (Elastic File System)**: マネージドなNFS。複数のAZから同時にマウント可能で、容量は自動拡張されます。

### 【詳細バリエーションと設定】
- **S3 ストレージクラス**:
    - **Standard**: 頻繁なアクセス用。
    - **Intelligent-Tiering**: アクセスパターンが不明な場合、自動で階層移動。
    - **Standard-IA (Infrequent Access)**: 低頻度アクセス。取り出し料金がかかる。
    - **One Zone-IA**: 1つのAZにのみ保存。コストは安いが耐久性に欠ける。
    - **Glacier Flexible Retrieval / Deep Archive**: アーカイブ用。取り出しに数分〜数時間かかる。
- **EBSのタイプ**:
    - **gp3 (General Purpose)**: 汎用SSD。コスパ最強。
    - **io2 (Provisioned IOPS)**: 高いIOPSが必要なデータベース等。
    - **st1 (Throughput Optimized HDD)**: ログ解析などスルーパス重視。

## 第3幕：データベース 〜フェイルオーバーは静かにやってくる〜

### 【スキット】
**佐藤**: 「昨夜、DBサーバーのメモリが死んでな。予備機に切り替えるのに2時間かかったよ。同期レプリの設定が甘かった……」
**田中**: 「お疲れ様です。RDSのマルチAZ設定なら、数分で勝手にフェイルオーバー終わってましたよ。DNSの書き換えまで自動です。あなたが寝汗をかいている間に、AWSが裏で勝手に掃除しておいてくれるんです」
**佐藤**: 「DNSの書き換えまで？ まさか、俺の仕事を奪う気か」
**田中**: 「仕事の中身が変わるだけです。物理的な修復じゃなくて、どのクラスのインスタンスを使うか、AuroraにするかDynamoDBにするかを考えるのが今のあなたの仕事です」
**佐藤**: 「オーロラ？ 北極にでも行くのか？」
**田中**: 「Amazon Auroraですよ。MySQL/PostgreSQL互換で、ストレージが6つのコピーを保持する化け物です。オンプレでこれと同じ構成を組もうとしたら、予算承認が降りる前に定年退職を迎えますよ」

### 【仕組み：どう動いているか】
- **Amazon RDS**: OSやDBエンジンのインストール、パッチ適用、バックアップを自動化するマネージドサービス。
- **Multi-AZ**: 別のAZにスタンバイ機を配置し、同期レプリケーションを行う。プライマリ障害時に自動でフェイルオーバー。
- **Amazon Aurora**: クラウド最適化されたストレージエンジンを持つ。データは3つのAZに計6つのコピーが保存され、非常に高い耐障害性とパフォーマンスを誇る。

### 【詳細バリエーションと設定】
- **データベースエンジンの選択**:
    - **RDS**: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server。
    - **Aurora**: MySQL/PostgreSQL互換。
    - **DynamoDB**: キーバリューストア（NoSQL）。ミリ秒単位の応答。
    - **ElastiCache**: Redis/Memcached（インメモリ）。
- **リードレプリカ**: 読み取り専用のコピー。読み取り負荷の分散に使用。非同期レプリケーション。

## 第4幕：ネットワーク 〜境界線は論理の中に〜

### 【スキット】
**佐藤**: 「田中、このシステム、社外からアクセスできないんだが。ルーターの設定ミスか？」
**田中**: 「佐藤さん、物理ルーターはここにはありません。VPCのセキュリティグループとネットワークACLを見ましたか？ あと、インターネットゲートウェイ（IGW）は付いてます？」
**佐藤**: 「セキュリティグループ？ ファイアウォールのことか」
**田中**: 「似てますが違います。セキュリティグループは『インスタンス単位』の用心棒。ネットワークACLは『サブネット単位』の検問所です。あと、プライベートサブネットからパッチをダウンロードしたいなら、NATゲートウェイを置いてください。外からは入れないけど、中からは出られる。ツンデレ仕様です」
**佐藤**: 「……物理的な配線がないと、どうも安心できんな」

### 【仕組み：どう動いているか】
- **VPC (Virtual Private Cloud)**: AWSクラウド内に論理的に隔離されたネットワーク。
- **Security Group**: ステートフルなファイアウォール（戻りの通信は自動許可）。
- **Network ACL**: ステートレスなファイアウォール（戻りの通信も明示的な許可が必要）。
- **NAT Gateway**: プライベートサブネットのインスタンスがインターネットにアクセスするために必要。

### 【詳細バリエーションと設定】
- **VPC エンドポイント**:
    - **Interface Endpoint**: PrivateLinkを利用。多くのサービスに対応。
    - **Gateway Endpoint**: S3とDynamoDB専用。無料。
- **Route 53**: 高可用なDNSサービス。フェイルオーバールーティングやレイテンシールーティングが可能。
- **CloudFront**: CDN（コンテンツ配信ネットワーク）。エッジロケーションを使用して、世界中に低レイテンシでコンテンツを配信。

## 第5幕：セキュリティと運用 〜最小権限の美学〜

### 【スキット】
**佐藤**: 「全部の権限を俺のアカウントにくれ。いちいち許可設定するのが面倒だ。 rootのパスワードを付箋で貼っておく方が効率的だろ」
**田中**: 「一番怖いこと言わないでください。それは『全裸で戦場を歩く』ようなもんです。IAM（Identity and Access Management）の基本は最小権限。必要な時に必要な分だけ。Rootアカウントは鍵をかけて地下深くに埋めておいてください」
**佐藤**: 「……じゃあ、この大量のログはどうすればいい？ 監査がうるさくてな」
**田中**: 「CloudTrailとConfigにお任せを。誰がいつ何をしたか、リソースがどう変わったか、全部記録されてます。あなたが隠れてインスタンスのサイズを上げたことも、バレバレですよ」
**佐藤**: 「（ギクリ）」

### 【仕組み：どう動いているか】
- **IAM**: ユーザー、グループ、ロール、ポリシーで構成。ロールをEC2等に付与（アサイン）することで、アクセスキーを直接持たせずに安全に権限を渡す。
- **AWS KMS (Key Management Service)**: データの暗号化に使用する鍵を管理。
- **AWS CloudTrail**: AWSアカウント内で行われたAPIコールの履歴を記録。
- **AWS Config**: リソースの構成変更を記録し、ルールへの準拠を判定。

### 【詳細バリエーションと設定】
- **IAMポリシー**:
    - **管理ポリシー**: AWSが用意したもの（職務別など）、またはユーザーが作成したもの。
    - **インラインポリシー**: 特定のエンティティに直接埋め込まれたもの。
- **モニタリング**:
    - **CloudWatch Metrics/Logs/Alarms**: リソースの監視と通知。
    - **Trusted Advisor**: コスト削減、セキュリティ、パフォーマンス向上のためのアドバイス。
- **AWS WAF / Shield**: Webアプリケーションへの攻撃防御と、DDoS保護。
- **AWS Organizations / SCP**: 複数アカウントの一元管理と、サービス利用制限。

## 第6幕：グローバル展開 〜世界を低レイテンシで繋ぐ〜

### 【スキット】
**佐藤**: 「田中、海外拠点からのアクセスが遅いって苦情が来てる。アメリカに中継サーバーを立てるか？」
**田中**: 「佐藤さん、また物理で解決しようとしてますね。CloudFrontを使ってキャッシュを世界中のエッジにばら撒けばいいんです。あと、IPアドレスを固定したいならGlobal Acceleratorを使ってください。AWSの専用ネットワークを通って、世界中から爆速でパケットが届きます。あなたが海底ケーブルを自作するより100万倍速いです」
**佐藤**: 「……海底ケーブルまでAWSが持ってるのか」

### 【仕組み：どう動いているか】
- **Amazon CloudFront**: 世界各地のエッジロケーションを使用して、静的・動的コンテンツを低レイテンシで配信。
- **AWS Global Accelerator**: AWSのグローバルネットワークを使用して、ユーザーからアプリケーションへのパスを最適化。

### 【詳細バリエーションと設定】
- **Global Accelerator**: 2つの固定エニーキャストIPを提供。
- **S3 Transfer Acceleration**: 長距離のS3アップロードを高速化。

## 第7幕：アプリケーション統合 〜疎結合が世界を救う〜

### 【スキット】
**佐藤**: 「田中、注文処理が詰まっていて、Webサーバーまで道連れに死んでるぞ。リクエストが多すぎるんだ」
**田中**: 「佐藤さん、密結合の末路ですね。Webサーバーから直接処理を呼ばないで、SQSのキューに放り込んでおけばいいんです。処理側が死んでても、メッセージはキューの中で大人しく待っててくれます」
**佐藤**: 「……行列に並ばせておくのか。それならWebサーバーはすぐ解放されるな」
**田中**: 「そうです。さらにLambdaを使えば、メッセージが来た時だけ勝手に起動して処理してくれます。サーバーを24時間監視して『生きてるか？』って確認する苦行から解放されるんですよ。サーバーレス、最高でしょ？」

### 【仕組み：どう動いているか】
- **Amazon SQS (Simple Queue Service)**: メッセージを保持するマネージド型キュー。送信側と受信側の処理速度の差を吸収する。
- **Amazon SNS (Simple Notification Service)**: メッセージを複数のサブスクライバー（Lambda, メール等）に一斉配信する（Pub/Sub）。
- **AWS Lambda**: サーバーを管理せずにコードを実行できる。イベント（S3へのファイル保存、SQSへのメッセージ着信など）をトリガーに起動する。

### 【詳細バリエーションと設定】
- **SQSの種類**:
    - **標準キュー**: 高いスループット、順序は保証されない（ベストエフォート）、少なくとも1回の配信。
    - **FIFOキュー**: 順序を厳守、重複配信なし。スループットに制限あり。
- **Step Functions**: 複数のLambdaやAWSサービスを繋いで、複雑なワークフローを視覚的に構築。

## 第8幕：移行と分析 〜データは新たな石油か、それとも重荷か〜

### 【スキット】
**佐藤**: 「田中、オンプレにある100TBのデータをクラウドに上げたいんだが、会社の回線が細くて終わらん。あと10年はかかりそうだ」
**田中**: 「まさか、光回線で100TB送ろうとしてます？ AWSから『雪玉（Snowball）』を呼んでください。物理的なハードディスクが箱に入って届きますから、それに詰めて送り返すだけです。結局、一番帯域が太いのは『ハードディスクを積んだトラック』なんですよ。古き良き物理の力です」
**佐藤**: 「……結局、最後は物理なのか。で、上げた後のデータはどうする？ SQLで適当に分析したいんだが」
**田中**: 「S3にあるデータなら、Athenaを使ってください。サーバーを立てずに、標準SQLを投げるだけで分析できます。もしテラバイト級のガチなデータウェアハウスがほしいならRedshiftですが、まずはAthenaで十分。あと、データ整形が面倒ならGlueっていう接着剤もあります」
**佐藤**: 「スノーボールにアテナ、グルー……。お前ら、名前をつけるセンスが独特すぎるだろ」

### 【仕組み：どう動いているか】
- **AWS Snow Family**: インターネット経由では時間がかかりすぎる大規模データ移行のための物理デバイス。
- **Amazon Athena**: S3上のデータに対して標準SQLを実行できるサーバーレスのインタラクティブなクエリサービス。
- **AWS Glue**: 完全マネージド型のETL（抽出・変換・ロード）サービス。

### 【詳細バリエーションと設定】
- **移行サービス**:
    - **Snowcone / Snowball Edge / Snowmobile**: 規模に応じた物理デバイス。
    - **AWS DMS (Database Migration Service)**: 稼働中のDBを最小のダウンタイムで移行。
- **分析サービス**:
    - **Amazon Redshift**: 大規模データ分析のための列指向データベース。
    - **Amazon Kinesis**: リアルタイムストリーミングデータの収集・処理。
    - **Amazon QuickSight**: マネージドなBI（ビジネスインテリジェンス）ツール。

## 結びに：エンジニアの心得
クラウドは、あなたがオンプレで苦労していた「重労働（Undifferentiated Heavy Lifting）」をAWSに押し付けるためのツールです。
佐藤さんのようなベテランの知識は、物理層から論理層、あるいはアーキテクチャ層へと昇華される必要があります。
さあ、コンソールを開いて、新しい「家畜」たちをデプロイしに行きましょう。
もちろん、最小権限でね。
